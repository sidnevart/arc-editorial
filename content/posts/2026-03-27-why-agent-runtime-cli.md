# Пост: зачем вообще нужен Agent Runtime CLI

## Текст поста

Большая проблема работы с ИИ в разработке не в том, что модели слабые. Проблема в том, что процесс обычно хаотичный.

Сегодня ты просишь “объясни”, завтра “сделай за меня”, послезавтра пытаешься вспомнить, что именно агент уже проверял и какие допущения придумал на ходу.

Из-за этого:

- теряется контекст
- растёт цена ошибок
- у пользователя деградирует мышление
- каждый новый заход начинается почти с нуля

Поэтому я делаю `arc` — local-first CLI runtime для Claude и Codex.

Идея простая: не просто “чат с моделью”, а управляемый runtime с режимами:

- `study` — чтобы агент учил, а не решал за тебя
- `work` — чтобы усиливал тебя как инженера
- `hero` — чтобы автономно закрывал bounded задачи, но с guardrails

Самая важная часть здесь даже не провайдеры. Самая важная часть — это discipline:

- deterministic pipeline
- approval gates
- layered memory
- context packaging
- reproducible artifacts после каждого run

Если делать agent tooling серьёзно, то именно это и важнее всего.

Не “ещё один AI wrapper”, а нормальная операционная среда для работы с агентами.

## Короткая версия

`arc` — это попытка превратить работу с ИИ из хаотичного диалога в управляемый инженерный процесс.

## Промпт для картинки

Нарисуй editorial-style иллюстрацию для статьи о local-first AI runtime для разработчиков.
Композиция: рабочий стол инженера, терминал в центре, вокруг три режима работы с подписями study, work, hero.
Вокруг терминала визуально показаны контекст, память, проверки и артефакты как структурированные слои.
Стиль: современный tech editorial, чистый, тёплые песочные и тёмно-зелёные цвета, без киберпанка, без фиолетового.
Формат: горизонтальная обложка 16:9, много воздуха, ощущение продуманной инженерной системы, а не футуристической магии.
