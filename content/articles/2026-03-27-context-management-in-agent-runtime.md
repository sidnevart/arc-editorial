# Статья: context management в agent runtime важнее, чем кажется

## Тезис

Когда обсуждают AI tooling для разработки, разговор почти всегда уходит в модели: какая сильнее, какая дешевле, какая быстрее. Но в реальной инженерной работе очень быстро становится видно, что качество системы определяется не только моделью. Его не меньше определяет то, как именно система управляет контекстом.

Именно context management отвечает за то, будет ли агент:

- действовать по существу
- выдумывать детали
- забывать ранее найденные факты
- тратить токены на шум
- смешивать человеческие правила и машинные индексы в одну кашу

`arc` как раз строится вокруг идеи, что runtime для агентов должен управлять контекстом осознанно.

## Почему “просто дать больше контекста” не работает

Интуитивный подход выглядит так: возьмём побольше файлов, подадим побольше текста, добавим память, вложим несколько документов и пусть модель сама разберётся.

На практике это почти всегда приводит к трём проблемам.

### 1. Шум становится дороже пользы

Когда модель получает слишком много всего сразу, она не становится точнее автоматически. Она получает больше материала, из которого можно случайно вытащить нерелевантную деталь.

### 2. Пропадает граница между фактом и гипотезой

Если в одном и том же пакете лежат старые заметки, человеческие TODO, машинные summaries и куски кода без явного разделения, модель начинает обращаться с этим как с одним уровнем достоверности.

### 3. Пользователь перестаёт понимать, откуда взялся ответ

Это уже не только техническая проблема. Это проблема доверия. Если нельзя быстро восстановить, какие источники были поданы и что из этого было реально проверено, система начинает деградировать как инженерный инструмент.

## Что я считаю правильной моделью

Для agent runtime нужен не “большой context blob”, а структурированная система.

Минимально полезная схема выглядит так:

1. Task brief.
2. Mode policy.
3. Human-authored project guidance.
4. Repo map и docs map.
5. Structured memory.
6. Машинный индекс.
7. Query-driven retrieval.
8. Compaction после этапов.

Ключевой момент здесь в том, что это не просто список источников. Это иерархия с разным весом и разной стабильностью.

## Как это выражено в `arc`

В текущем MVP есть уже несколько важных компонентов.

### Layered memory

Память хранится не одним файлом, а как structured entries с полями:

- `id`
- `scope`
- `kind`
- `source`
- `confidence`
- `created_at`
- `last_verified_at`
- `status`
- `tags`

Это позволяет хотя бы не путать decision, question и fact.

### Human-authored maps отдельно от machine index

Это очень важное архитектурное решение. Человеческие карты проекта нужны для устойчивого смысла. Машинный индекс нужен для поиска и дешёвого retrieval. Эти два слоя не должны сливаться.

### Query-driven context packaging

Новая итерация `arc` уже уходит от полностью статического context pack и собирает:

- query terms
- identifier candidates
- relevant docs
- relevant code surfaces
- grep hits
- optional AST hits
- relevant memory

То есть вопрос теперь не только “какие источники есть”, а “что именно из них относится к этой задаче”.

## Почему `rg` и `ast-grep` здесь так важны

Если у тебя нет дешёвого локального retrieval, ты почти неизбежно начнёшь компенсировать это большим prompt. А большой prompt — это дорого, хрупко и плохо управляется.

`rg` хорош как fast lexical search.

`ast-grep` хорош как более структурный шаг поверх кода, когда важно искать уже не просто строку, а кодовые формы и конструкции.

Для меня это и есть правильный путь:

- сначала дешёвый локальный поиск
- потом компактный retrieval
- и только потом модель

## Где тут всё ещё слабые места

Было бы нечестно делать вид, что вопрос закрыт.

Сейчас в `arc` context management уже заметно лучше, чем просто “склеить несколько markdown-файлов”, но до сильного production-grade retrieval layer ещё далеко.

Остаётся сделать:

- более умный scoring релевантности
- свежесть уровня `fresh/warm/stale`
- selection не только по словам задачи, но и по proximity к изменяемым файлам
- нормальный retrieval loop перед human handoff
- более точную работу с docs excerpts

## Вывод

Agent runtime выигрывает не тогда, когда в него встроили “самую умную модель”. Он выигрывает тогда, когда умеет подавать модели ровно тот контекст, который нужен для текущего решения, и при этом сохранять следы своей работы.

Именно поэтому context management — это не второстепенная часть агентной системы. Это и есть одна из её центральных инженерных задач.
