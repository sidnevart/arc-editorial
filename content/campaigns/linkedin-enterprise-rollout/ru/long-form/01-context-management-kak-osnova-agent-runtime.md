# Long-Form 01: context management как основа agent runtime

## Подводка

Когда говорят про AI tools для разработки, почти всегда обсуждают модель. Но на практике устойчивость системы определяется не только и не столько моделью. Она определяется тем, как устроен контекст.

## Текст

За последние месяцы мне всё чаще кажется, что главное инженерное поле в AI tooling лежит не в выборе лучшей модели, а в проектировании правильного context management.

Почему это так важно.

Если у системы нет хорошего context management, даже очень сильная модель начинает:

- тащить шум в решение
- смешивать факты и предположения
- опираться на устаревшие куски знания
- расходовать слишком много токенов на нерелевантный материал

И наоборот, когда runtime умеет:

- разделять human-authored guidance и machine-generated index
- держать structured memory
- собирать query-driven context pack
- давать model только релевантные docs, code surfaces и search hits

тогда поведение системы становится заметно более взрослым.

Это особенно важно в enterprise, где AI tool должен быть не просто helpful assistant, а частью инженерного процесса.

В моём случае ключевым оказался переход от статичного context blob к более осмысленному pack:

- task brief
- mode policy
- project guidance
- repo map
- docs map
- relevant memory
- search hits через `rg`
- structural hits через optional `ast-grep`

Этот сдвиг дал не только более аккуратную подачу контекста, но и заметный operational эффект.
В ряде внутренних benchmark-задач стало видно до 2x снижения token consumption без деградации качества.

Но ещё важнее другое: система стала меньше “говорить не по делу”.

Именно поэтому я всё сильнее уверен:

будущее AI dev tools — не в бесконечном увеличении prompt size.
Оно в точности и управляемости context packaging.
