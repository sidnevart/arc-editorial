# Статья: почему deterministic pipeline лучше хаотичного agent loop

## Проблема

У многих agent systems внутренняя логика выглядит примерно так: модель что-то думает, что-то ищет, иногда вызывает инструменты, потом снова что-то думает, а пользователь по сути видит только финальный ответ или поток шагов без устойчивой структуры.

Это может выглядеть впечатляюще, но инженерно создаёт проблемы:

- сложно воспроизводить поведение
- сложно отлаживать сбои
- сложно понять, на каком этапе возникла ошибка
- сложно отделять planning от implementation
- сложно строить governance и quality gates

Для production-подобного инструмента этого недостаточно.

## Что даёт deterministic pipeline

В `arc` базовая идея другая: оркестрация должна быть state machine.

Не “бесконечный intelligent loop”, а явные состояния:

- `initialized`
- `context_collecting`
- `planning`
- `implementing`
- `verifying`
- `reviewing`
- `documenting`
- `done`
- `failed`
- `blocked`

Эта модель сразу даёт несколько преимуществ.

### 1. Поведение становится объяснимым

Можно посмотреть run и понять не просто “что ответил агент”, а где именно он находился в процессе.

### 2. Failures становятся локализуемыми

Если провайдер умер, это уже не абстрактная ошибка “что-то пошло не так”. Это конкретно failure implementation stage или provider execution.

### 3. Проще встраивать guardrails

Approval gate ставится до provider execution. Verification идёт после implementation. Review идёт после verification. Это не рекомендация, а структура процесса.

### 4. Артефакты привязываются к этапам

Когда pipeline явный, артефакты тоже становятся осмысленными:

- planning artifacts
- implementation log
- verification report
- review report
- docs delta

## Почему это особенно важно для AI-инструментов

С ИИ очень легко построить систему, которая “кажется умной”, но плохо операционализируется.

В demo это почти незаметно.

В реальной работе сразу становится важно:

- что именно система делала
- где она остановилась
- что проверила
- что осталось неизвестным
- что можно перезапустить

Deterministic pipeline как раз отвечает на эти вопросы.

## Это не ограничение, а форма дисциплины

Иногда кажется, что жёсткая state machine делает агент “менее гибким”.

На деле это не так.

Гибкость остаётся внутри этапов, но общая форма процесса становится управляемой.

Это очень похоже на хорошую инженерную практику вообще:

- свобода внутри bounded step
- жёсткость на уровне interfaces и lifecycle

## Вывод

Если строить agent runtime не как игрушку, а как рабочий инструмент, deterministic pipeline почти неизбежно оказывается лучше хаотичного loop.

Он делает систему:

- понятнее
- проверяемее
- безопаснее
- удобнее для отладки
- лучше пригодной для команды, а не только для автора демо

Именно поэтому для `arc` state machine — не внутренняя реализационная деталь, а одно из главных архитектурных решений.
